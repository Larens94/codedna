# CodeDNA Benchmark — Methodology & Test Preparation

## Overview

This benchmark measures how CodeDNA annotations affect an AI agent's ability to **locate the correct files** for a code modification task. It compares two conditions:

- **Control**: agent navigates a vanilla Django codebase (no annotations)
- **CodeDNA**: agent navigates the same codebase with CodeDNA v0.7 annotations

Both conditions use the same LLM, tools, file size limits, and temperature.

---

## Test Preparation

### 1. Task Selection

Tasks are selected from [SWE-bench](https://www.swebench.com/), a public benchmark of real GitHub issues. We selected 5 multi-file Django tasks where the ground truth requires modifying 7-10 files across multiple packages.

### 2. CodeDNA Annotation (codedna variant only)

The codedna variant has three layers of annotation:

#### a) `.codedna` manifest (project root)
Auto-generated by `codedna init`. Describes the package tree, purposes, and inter-package dependencies.

#### b) Module headers (every Python file)
Auto-generated by the CodeDNA annotator. Each file gets a docstring header with:
- `exports:` — public API symbols
- `used_by:` — files that import from this module (the navigation graph)
- `rules:` — architectural constraints

#### c) Architectural rules (`rules:` field)
Written by developers with knowledge of Django's architecture, simulating the knowledge accumulation that occurs through repeated use of the codebase. **Rules are not task-specific** — they describe real architectural constraints that would be true regardless of which task is being evaluated.

> **Transparency note**: The `rules:` field was populated for **51 files** across the entire `django/db/` subtree, not just the 7 ground truth files. Rules describe general Django patterns (e.g., "SQLite has no ALTER COLUMN; schema changes recreate the table") and do not mention XOR or any specific task.

### 3. Control Variant

The control variant uses the exact same Django codebase but **without any CodeDNA annotations** — no `.codedna` manifest, no module headers, no rules. Files are plain Django source code.

### 4. Agent Configuration

| Parameter | Value |
|-----------|-------|
| READ_FILE_LIMIT | 20,000 characters |
| GREP_LIMIT | 4,000 characters |
| Max turns | 30 |
| Temperature | 0 |
| Tools | `list_files`, `read_file`, `grep` |

Both control and codedna agents receive a system prompt describing the task and available tools. The codedna agent additionally receives instructions on how to parse CodeDNA annotations (generic, no task-specific hints).

---

## Metrics

- **F1 (read)**: harmonic mean of recall and precision, computed on **files actually read** by the agent vs ground truth files
- **F1 (proposed)**: same metric but on files the agent **proposes to modify** in its final response
- **Token consumption**: total characters read across all `read_file` calls

Primary metric: **F1 (read)** — it measures pure navigation efficiency without penalizing the agent for mentioning files it didn't read.

---

## Preliminary Results (1 task, 2 models)

| Model | Control F1 | CodeDNA F1 | Δ F1 | Token Savings |
|-------|-----------|------------|------|---------------|
| Gemini 2.5 Flash | 43% | 67% | **+24%** | -33% |
| GPT-5.3 Codex | 40% | 55% | **+15%** | +32% |

> **Note**: These are preliminary results on a single task (django__django-14480). Statistical significance requires completing all 5 tasks with multiple models.

---

## Limitations & Disclaimers

1. **Sample size**: N=1 task is insufficient for statistical claims. Results are indicative, not conclusive.
2. **Rules authorship**: `rules:` were written by developers with Django architecture knowledge, simulating the knowledge accumulation that occurs through repeated use of a codebase. They were not written to favor specific tasks, but describe real architectural constraints.
3. **Annotation effort**: CodeDNA annotations require initial setup time. This cost is not measured in the benchmark but is a real consideration for adoption.
4. **Single domain**: All tasks are from Django (Python web framework). Generalization to other languages/frameworks is not tested.
