Il problema con i sistemi esistenti

📄

CLAUDE.md — Statico e Verboso

L'utente deve scrivere e mantenere manualmente il file di contesto. Cresce nel tempo e intasa il prompt con centinaia di token irrilevanti.

🔍

RAG / Vector DB — Latency

Il contesto viene cercato al momento della query. Dipende dalla qualità degli embeddings e introduce latenza ad ogni richiesta.

🧠

MemGPT — Overhead architetturale

Simula una gerarchia di memoria RAM/disco. Complesso da mantenere, richiede infrastruttura aggiuntiva e può paginare contesto rilevante.

Architettura

Come vede il modello

Il modello legge il file dall'alto verso il basso. Il manifest arriva prima del codice. Quando vede DEPENDS_ON sa già cosa non deve rompere.

main.py — con manifest Beacon 
# ============================================================
# FILE: main.py
# PURPOSE: Dashboard KPI fatturato mensile con grafico e tabella
# DEPENDS_ON: utils.py → calcola_kpi(), format_currency()
# EXPORTS: render(execute_query_func) → HTML string
# STYLE: tailwind, chartjs
# DB_TABLES: ordini (mese, fatturato, costo)
# LAST_MODIFIED: aggiunta colonna margine alla tabella
# ============================================================

# ← Il modello ha già tutto il contesto. Ora legge il codice con consapevolezza.

from .utils import calcola_kpi, format_currency

def render(execute_query_func):
    rows = execute_query_func("SELECT mese, fatturato, costo FROM ordini")
    kpi  = calcola_kpi(rows)
    # → da utils.py: calcola_kpi gestisce normalizzazione e margini
    ...
1

Planner decide la struttura

Fase 1: l'AI riceve il prompt utente + schema DB + LiveCtx (contesto di progetto compresso in ~50 token Python). Decide quanti file generare e con quale scopo.

2

Executor genera ogni file con manifest

Fase 2: per ogni file, l'AI scrive prima il manifest (PURPOSE, DEPENDS_ON, EXPORTS, STYLE, DB_TABLES, LAST_MODIFIED) poi il codice. Il file è self-describing dalla prima riga.

3

Editor modifica con piena consapevolezza

In modalità edit, l'AI legge il manifest in testa al file prima ancora del codice. Produce blocchi SEARCH/REPLACE chirurgici. Il primo blocco aggiorna sempre LAST_MODIFIED.

4

LiveCtx apprende in background

Dopo ogni generazione, un micro-call AI estrae fatti strutturati (CSS usato, chart library, preferenze UX) e li salva come Python typed literal in memoria. La prossima richiesta trova già il contesto pronto — zero latency.

Standard

File Manifest Spec

Ogni campo nel manifest ha una regola precisa che massimizza la densità informativa per token.

📄 FILE

  • Nome esatto del file
  • Usato dal Planner per leggere solo i manifest (prime 15 righe)
  • Mai abbreviare o omettere

🎯 PURPOSE

  • 1 riga, max 12 parole
  • Descrive il COSA, non il COME
  • Es: "Dashboard KPI fatturato con grafico e tabella"

🔗 DEPENDS_ON

  • Formato: file.py → func1, func2
  • Lista ogni import esterno al file
  • "nessuno" se non ci sono dipendenze

📤 EXPORTS

  • Funzioni e classi pubbliche con firma
  • Es: render(execute_query_func) → str
  • Aiuta il modello a capire cosa non deve rimuovere

🎨 STYLE

  • CSS framework e chart library
  • Es: tailwind, chartjs
  • Garantisce coerenza visiva tra file dello stesso progetto

📝 LAST_MODIFIED

  • Ultimo edit in max 8 parole
  • Aggiornato ad ogni modifica (primo SEARCH/REPLACE block)
  • Tiene lo storico integrato nel file

Roadmap

Stato del framework

✓ Done

File Manifest Header — generazione automatica

Ogni file generato dall'AI inizia con il blocco manifest. L'Executor è istruito a compilarlo con i dati reali del file.

✓ Done

LAST_MODIFIED — aggiornamento ad ogni edit

In modalità Edit, il primo SEARCH/REPLACE block aggiorna il campo LAST_MODIFIED del manifest. Il file tiene lo storico integrato.

✓ Done

LiveCtx — contesto di progetto in-memory

Preferenze apprese automaticamente dal codice generato (CSS, chart lib, pattern UX) salvate in DB e in RAM. Zero retrieval latency.
