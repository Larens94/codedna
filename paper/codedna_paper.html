CodeDNA: An In-Source Communication Protocol
for AI Coding Agents

Fabrizio Corpora
Independent Research · fabrizio.corpora@gmail.com
Zenodo preprint · March 2026 · doi:10.5281/zenodo.19110474
Abstract LLM agents that autonomously edit production codebases suffer from context blindness: they modify a function without awareness of cross-file dependencies, producing silent bugs and broken cascades. Existing mitigations — RAG, project-level rules files (CLAUDE.md, AGENTS.md), or long-context injection — impose retrieval latency, maintenance overhead, or infrastructure cost. Recent work (ETH Zurich, 2026) shows that LLM-generated context files can degrade task success by 3% and increase inference costs by over 20%. We present CodeDNA, an in-source communication protocol that embeds dependency graphs, public contracts, agent constraints, and semantic naming conventions directly into source files. The standard is named after its biological parallel: just as DNA encodes the complete blueprint of an organism in every cell, CodeDNA ensures that every code fragment carries sufficient architectural context for an agent to act correctly in isolation. We evaluate CodeDNA with a multi-model SWE-bench evaluation across 3 LLMs (Gemini 2.5 Flash, DeepSeek Chat, Gemini 2.5 Pro) on 5 real Django issues, 5 runs/task at T=0.1 — CodeDNA improved file localization F1 on all 3 completed models, with a statistically significant gain of +13 percentage points on Gemini 2.5 Flash (60% → 72%, Wilcoxon p=0.040), +9pp on DeepSeek Chat (50% → 60%), and +9pp on Gemini 2.5 Pro (60% → 69%). These results measure navigation efficiency (which files the agent reads), not end-to-end patch correctness, isolating the navigation bottleneck that CodeDNA specifically addresses. CodeDNA requires no build step, no external system, and zero retrieval latency.

1. Introduction

The adoption of LLM coding agents in production workflows has exposed a class of failure that single-file benchmarks routinely miss: cross-file context blindness. When an agent reads a function in isolation — because it operates in sliding-window mode or because the relevant contract resides in another module — it cannot observe the architectural constraints governing correct behaviour.

SWE-bench [1], the field's principal multi-file benchmark, demonstrated in 2024 that state-of-the-art models could resolve only 1.96% of real GitHub issues. The dominant failure mode was not reasoning quality but navigation cost: finding the right files required too many tool interactions. Subsequent work confirms the pattern: LoCoBench-Agent [2] (2025) measures tool-usage efficiency in long-context engineering workflows and finds systematic over-exploration; PRDBench [3] (2025) reports that agents spend the majority of their token budget on navigation rather than repair.

Current mitigation approaches share a structural weakness: context lives outside the code. CLAUDE.md and CursorRules are session-level files invisible to agents in sliding-window mode. RAG [6] systems introduce retrieval latency and require embedding infrastructure. A 2026 ETH Zurich study [4] found that LLM-generated context files (AGENTS.md) degraded task success by 3% and increased costs by over 20%.

We propose a complementary approach: embed context inside the code, at every level, so that any fragment an agent reads is already self-sufficient.

2. Related Work

2.1 Multi-File Agent Benchmarks

SWE-bench [1] (Jimenez et al., 2024) evaluates LLMs on 2,294 real GitHub issues across 12 Python repositories. It established that multi-file coordination is the principal bottleneck, with a 1.96% resolution rate at launch. CodeDNA addresses the navigation cost dimension that SWE-bench exposed.

LoCoBench-Agent [2] (November 2025) provides the closest methodological parallel to our work: it benchmarks LLM agents in long-context software engineering scenarios with 8,000 tasks and measures tool-usage efficiency explicitly. Our metrics (tool calls per bug, file reads, grep frequency) directly parallel LoCoBench-Agent's efficiency framing.

PRDBench [3] (October 2025, presented at AAMAS 2026) uses 50 real Python projects and an Agent-as-a-Judge paradigm to evaluate agents from Product Requirement Document to implementation. It confirms that agents over-explore when codebase structure is opaque.

HumanEval [5] (Chen et al., 2021) measures single-function synthesis from docstrings. Its pass@k metric is complementary: it captures output quality but not cross-file navigation, making it orthogonal rather than competing.

2.2 Context Management for LLM Agents

Retrieval-Augmented Generation [6] retrieves relevant chunks from a vector store at inference time. Applied to code, retrieval quality depends on embedding similarity, which may not align with architectural relevance and requires additional infrastructure.

Rules files (CLAUDE.md [10], CursorRules, AGENTS.md, GitHub Copilot instructions [9]) impose project-level conventions but are invisible to sliding-window agents and require manual maintenance. The ETH Zurich study [4] provides the strongest evidence against the external-file approach: even carefully crafted human-written files offer only a 4% gain, and LLM-generated ones actively degrade performance by −3% while increasing costs by +20%.

MemGPT [7] maintains compressed memory across turns. This addresses recall of prior context but not forward navigation: a history of prior reads does not tell an agent what to read next. Knowledge graph–based approaches (Tongyi Lingma [11], 2025–2026) use graph traversal to expose cross-module call relationships but rely on external infrastructure and dynamic analysis. CodeDNA's key differentiator: the navigation graph is co-located with the code itself, requiring no retrieval system and surviving in any sliding window.

2.3 Code Annotation and Documentation Standards

Documentation standards (docstrings, JSDoc, Doxygen, Literate Programming [8]) focus on human readability and API documentation generation. They do not encode cross-file navigation semantics or agent-specific constraints. Type annotation systems (PEP 484, TypeScript) encode type contracts but not architectural ones. CodeDNA differs by designing annotations as a machine-readable navigation protocol targeting the failure modes of LLM agents rather than human readers.

3. The CodeDNA Communication Protocol v0.7

3.1 Biological Analogy

The name CodeDNA captures the standard's core design principle by analogy to molecular biology. Biological DNA encodes the complete blueprint of an organism in every cell: each cell's nucleus contains the full genome, not just the instructions for its own specialised function. This redundancy means that any cell — regardless of its location or context — can be read by the right machinery and yield architectural information about the whole organism.

CodeDNA applies the same principle to software: every source file encodes not only its own logic but also its position in the architectural organism — what it depends on, what depends on it, what contracts it must honour. The protocol comprises four levels (0–3): Level 0 is the project manifest (.codedna), operating at repository scale. Levels 1–3 operate at file, function, and variable scale respectively, and mirror three scales of biological encoding:

Table A. Biological DNA ↔ CodeDNA Analogy
Biological conceptCodeDNA equivalent
Full genome in every cellL1 module header in every file: full architectural position
Gene = local function unitL2 function docstring: local contract + cross-file deps
Codon = minimal encoding unitL3 inline comment at call site: 1-line context at danger point
Holographic redundancyAny 10-line extract carries enough context to navigate safely
No external lookup neededZero retrieval latency; context travels with the code

The holographic property — cutting a hologram in half produces two complete smaller images, not two halves — explains CodeDNA's sliding-window robustness: even when an LLM reads only lines 50–80 of a file, it encounters Level 2 function docstrings and Level 3 call-site comments that restate the critical architectural rules at the exact point of risk.

3.2 Level 0 — Project Manifest

A single YAML file (.codedna) at the repository root. The agent reads this first — before opening any source file — to understand package purposes and inter-package dependencies. It is injected into the agent's initial context so the full structural map is available before any tool call is made.

project: myapp
packages:
  payments/:
    purpose: "Invoice generation, payment processing"
  analytics/:
    purpose: "Revenue reports, KPI dashboards"
    depends_on: [payments/, tenants/]
  tenants/:
    purpose: "Multi-tenant management, suspension"

The manifest is the entry point for the Planner Read Protocol: (1) read .codedna; (2) read only the module docstring of each candidate file (first 8–12 lines); (3) build an exports: → used_by: graph; (4) open only files where the graph intersects with the task. This pattern reduces unnecessary reads while maximising coverage of architecturally relevant files.

3.3 Level 1 — Module Header

A Python-native module docstring that must be the first content in every file. It encodes the full architectural position of the file in approximately 70 tokens.

"""analytics/revenue.py — Monthly revenue aggregation.

deps:    payments/models.py → get_invoices_for_period
         tenants/models.py → is_suspended
exports: monthly_revenue(year,month)->dict
used_by: api/reports.py → revenue_route
tables:  invoices(tenant_id, amount_cents, status)
         tenants(suspended_at, deleted_at)
rules:   get_invoices_for_period() returns ALL tenants,
         NO suspended filter → MUST call is_suspended()
         BEFORE aggregating revenue
"""

Using Python's native triple-quote string format (rather than a custom comment block) maximises LLM comprehension: models trained on billions of Python files apply existing pattern-matching to the docstring's structure, rather than parsing an unfamiliar protocol from scratch.

The rules field encodes hard constraints that apply to every edit in the file. Unlike inline annotations (function-scoped), rules applies globally, acting as the cell's genome: available anywhere in the file without requiring the reader to navigate to a specific function.

3.4 Level 2 — Sliding-Window Annotations

Agents frequently extract 50–100 lines from the middle of a file, bypassing the module header. Level 2 embeds navigation cues at the function level via two sub-layers.

Level 2a — Function Docstring (Google style):

def monthly_revenue(year: int, month: int) -> dict:
    """Aggregate paid invoices into monthly totals.

    Depends: payments.models.get_invoices_for_period
             — returns ALL invoices, NO suspended filter.
    Rules:   MUST filter is_suspended() BEFORE summing.
    """

Level 2b — Call-site inline comment (last line of defence):

    # includes suspended tenants — filter below
    raw = get_invoices_for_period(year, month)
    total = sum(i['amount_cents'] for i in raw)

The call-site comment ensures that even a 10-line window centred on the dangerous call carries the critical context. This is the codon-level encoding: the smallest unit that still conveys architectural meaning.

3.5 Level 3 — Semantic Naming

Variable names encode type, shape, domain, and origin. Format: <type>_<shape>_<domain>_<origin>. 

# ❌ Standard — agent must trace to understand
data  = get_users()
price = request.json["price"]

# ✅ CodeDNA — self-documenting in any 10-line window
list_dict_users_from_db  = get_users()
int_cents_price_from_req = request.json["price"]

4. Benchmark Design

We measure navigation efficiency — how many tool interactions an autonomous agent requires to correctly localize the root cause of a bug in a multi-file codebase.

4.1 Agent Implementation

We implemented an autonomous agent using Gemini 2.5 Flash (temperature 0) with three function-calling tools: read_file(path), list_files(directory), and grep(pattern, directory). No file contents are pre-loaded. The agent runs to completion (max 20 turns) or until it stops calling tools. While temperature 0 minimises sampling variance, we note that the full agent loop (tool orchestration, API behaviour) is not strictly deterministic in the mathematical sense. Metrics: total tool calls, read_file calls, grep calls, conversation turns, and found_correct (boolean — whether the identified root-cause file is architecturally correct).

5. Results

5.1 SWE-bench Multi-Model Evaluation

To test whether CodeDNA's benefit generalises across different LLM architectures, we evaluated 3 models on 5 real Django issues from SWE-bench. Each model received identical tools (read_file, list_files, grep), the same system prompt, and a maximum of 30 turns. Each condition was repeated 5 times per task at T=0.1 to obtain stable mean estimates; statistical significance is evaluated with the Wilcoxon signed-rank test (one-tailed, H1: CodeDNA > Control) over the 5 per-task mean F1 pairs. The primary metric is file localization F1: the harmonic mean of precision (fraction of files read that are in the ground-truth patch) and recall (fraction of ground-truth files that the agent actually read). We emphasise that this metric measures navigation accuracy — whether the agent reads the right files — not whether it produces a correct patch. Isolating the navigation component allows us to evaluate CodeDNA's specific value proposition without conflating it with the model's code synthesis ability.

Table 3. Multi-model file localization F1 — 5 SWE-bench tasks, 5 runs/task at T=0.1, Control vs. CodeDNA.

Model Ctrl F1 DNA F1 Δ F1 p-value Tasks Won
Gemini 2.5 Flash 60% 72% +13pp 0.040* 4 / 5
DeepSeek Chat 50% 60% +9pp 0.11 4 / 5
Gemini 2.5 Pro 60% 69% +9pp 0.11 3 / 5
Key result: All 3 models complete. Gemini 2.5 Flash achieves statistical significance (Wilcoxon W+=14, N=5, p=0.040, Δ=+13pp). DeepSeek Chat: Δ=+9pp, p=0.11. Gemini 2.5 Pro: Δ=+9pp, p=0.11. Gemini 2.5 Flash shows the strongest benefit. One hypothesis is that Flash's more concise reasoning style is better suited to following structured annotation chains than Pro's more verbose approach. Gemini 2.5 Pro ctrl=60% matches Flash, suggesting these tasks do not favour larger models in the control condition. The three models use different provider stacks (Google native function calling vs. DeepSeek's OpenAI-compatible endpoint), which introduces some cross-provider variability not fully controlled. This does not invalidate the comparison — the annotations themselves are identical — but the observed differences reflect both CodeDNA's effect and provider-specific runtime behaviour.

5.2 When CodeDNA Helps Most

Analysis across 5 SWE-bench tasks (Gemini 2.5 Flash, ≥5 runs each at T=0.1) reveals a clear pattern correlated with the structural type of the task:

Table 3a. Δ F1 by task type (Gemini 2.5 Flash).

Task type Task Δ F1
Clear dependency chain — A calls B delegates to C 12508 (dbshell) +9%
Delegation with backend fan-out — one interface, N backends 13495 (Trunc tzinfo) +22%
Feature addition with flag gating — across feature/schema layers 11991 (INCLUDE) +17%
XOR feature with multi-layer propagation 14480 (Q() XOR) +17%
Cross-cutting fix — same pattern in N independent files, no shared ancestor 11808 (__eq__) ≈0%

CodeDNA is most effective when there is a navigable call chain. The used_by: graph guides the agent from the entry point to all affected files. For cross-cutting concerns — the same fix in N independent files with no common ancestor — the benefit is near-zero because no navigation path connects them. Task 11808 was included deliberately to test the protocol's structural limits; Δ≈0% is reported as a known limitation, not hidden. See §5.3 for the annotator integrity audit and §6.6 for the proposed v0.8 extension (cross_cutting_patterns:).

Per-task qualitative analysis. The five tasks span a spectrum of structural difficulty. Task 12508 (dbshell) is the easiest to enter: the management command file is named after the feature; the challenge is discovering the four backend client files, which used_by: handles reliably (+9pp). Task 11991 (INCLUDE) follows a similar pattern — schema.py is findable by grep, and used_by: enumerates all four backend schema editors explicitly (+17pp). Task 14480 (Q() XOR) requires following a multi-layer chain — from query_utils.py through where.py, sql/__init__.py, and per-backend features.py — exactly the cascade that used_by: enables (+17–18pp). Task 13495 (Trunc tzinfo) is the most model-divergent: Flash gains +22pp while DeepSeek and Pro each regress by 8–9pp. The entry file (db/models/functions/datetime.py) is reachable for all models, but following used_by: chains to backend files appears to introduce a precision trap for DeepSeek and Pro — a structural anomaly under investigation. Task 11808 (__eq__) is the structural limit of v0.7: the entry point models/base.py is a 847-line generic file, and the five affected subclasses are independent with no shared caller. Flash control already reaches F1=0.58 — CodeDNA adds nothing (Δ=−1.5%). The same task is DeepSeek's strongest result (Δ=+34pp) because DeepSeek control achieves only F1=0.20: the model cannot locate the entry point without the structural hint in the module docstring's rules: field.

One observation from the multi-model data is that cheaper or less capable models appear to benefit more from CodeDNA than frontier models — a working hypothesis, not a confirmed finding. This observation constitutes a working hypothesis, not a confirmed finding: DeepSeek and Pro show positive but non-significant Δ (p=0.11), and a larger multi-model study is required to test the hypothesis directly.

5.3 Annotation Integrity Audit

A key concern for any navigation benchmark is whether annotations guide the agent to the right files by describing the architecture, or by encoding the solution. We performed a full audit of the benchmark annotations to verify no task-specific hints were embedded.

Methodology. For each task, every ground-truth file (from files_in_patch) was inspected. The used_by: targets were classified as GT (ground-truth) or non-GT. The rules: field was evaluated for task-specific prescriptions vs. architectural descriptions. The test: if an agent reads only the rules: field, does it still need to open files to determine what to fix? A correct annotation passes this test; one that pre-populates a file list fails.

Table 3b. Annotation integrity verdicts per task.

Task used_by: verdict rules: verdict
11808 (__eq__) Files independent — no used_by: chain by design Describes Python data model convention; Δ≈0% confirms no navigation advantage
14480 (XOR) 16 targets in query_utils.py; only 2 are GT, both genuine callers Describes connector system mechanics; agent must still reason which files are relevant
13495 (Trunc tzinfo) base/operations.py lists all 4 backends — all backends that exist _convert_field_to_tz() per-backend: accurate architecture, not a fix prescription
11991 (INCLUDE) Corrected during audit to include all 4 backend schema editors (see note) Architectural delegation pattern; no file list for the fix
12508 (dbshell) base/client.py lists all 4 backend clients — complete by definition Describes existing architecture accurately; no task-specific prescriptions possible

Annotation correction made during audit. django/db/backends/base/schema.py in task 11991 had an incomplete used_by: listing only postgresql/schema.py. All four backend schema editors (mysql, oracle, postgresql, sqlite3) genuinely inherit from BaseDatabaseSchemaEditor. The annotation was corrected to reflect the complete inheritance graph before benchmark runs.

Verdict. The benchmark annotations are architecturally accurate and do not encode task-specific solutions. Where GT files appear in used_by: targets, it is because those files are genuine callers or subclasses — not because the annotator cherry-picked them. The empirical results confirm: tasks with navigable chains show Δ>0%; task 11808 shows Δ≈0% regardless of rules: content.

5.5 Retry Cost Analysis

File-localization F1 is typically reported as an efficiency metric. We argue it should also be treated as a correctness proxy: an agent that reads fewer than half the ground-truth files is unlikely to produce a complete patch, forcing the user to re-invoke the entire session. We define a run as high-risk when per-run recall falls below 50%.

Across 131 multi-run trials (DeepSeek-Chat, Gemini 2.5 Flash, Gemini 2.5 Pro — 5 tasks each, 3–6 runs per task), the fraction of high-risk runs was:

Table 5. Incomplete-fix risk (recall < 50%) pooled across three models and five SWE-bench tasks.

Condition High-risk runs Total runs Risk rate
Control 34 66 52%
CodeDNA 16 65 25%
Reduction −27 pp (−52% relative)

The most pronounced case is DeepSeek-Chat on task 11808 (10 ground-truth files): all five control runs achieved recall of 10–20% (finding 1–2 of 10 files); all five CodeDNA runs achieved 30–60% recall. Every control run would have produced an incomplete patch. Under CodeDNA, three of five runs exceeded the 50% threshold.

Gemini 2.5 Pro provides a complementary data point: CodeDNA reduced mean tool calls by 14.3% while raising mean F1 by 9 pp, demonstrating that structured context enables the model to navigate directly to relevant files rather than exploring broadly — fewer calls, higher accuracy.

These results support an accuracy-first framing for annotation protocols: even if CodeDNA were cost-neutral in tool calls, halving the rate of incomplete navigation would justify its adoption by halving the rate of sessions that must be repeated.

6.2 Why In-Source Annotation Works

The mechanism mirrors the biological insight: when the contract governing a function is declared at the function, capable agents do not need to retrieve it from its original definition. When a rules field propagates a contract downstream to the exact location where it is violated, the agent can diagnose the issue correctly without following the dependency back to its original definition.

The ETH Zurich finding [4] that LLM-generated context files degrade performance precisely motivates this approach: if external context files cause over-exploration, the solution is not better files but embedded context that arrives with the code the agent was already going to read. CodeDNA turns every file read into a context delivery event.

6.3 Format Design Rationale

The choice of Python-native module docstrings (rather than custom comment blocks) for Level 1 was validated empirically: models trained on Python corpora apply existing pattern-matching to the docstring's deps: / exports: / rules: structure. The custom # === CODEDNA:0.5 === block format, while machine-parsable, required the model to process an unfamiliar syntax schema, yielding marginally longer reasoning chains before exploitation.

6.4 Annotation Overhead and the Agentic Workflow Framing

A common objection to annotation standards is the overhead of writing and maintaining them. For CodeDNA, this objection must be evaluated in context. CodeDNA is designed for agentic code generation workflows — environments in which an LLM agent both generates and modifies source files. In this setting, the agent itself produces the module headers and function docstrings as a natural byproduct of generation. The human developer does not annotate 500 files manually; the coding agent leaves its architectural memory in-place as it writes. The marginal cost of annotation approaches zero.

This framing also redefines what CodeDNA is: not a standard that asks developers to write more comments, but a convention that gives agents a structured format in which to record their own local architectural understanding. The standard converts agent-generated code from a passive output into an active, self-describing navigation graph for the next agent that reads it.

A secondary observation is that CodeDNA's structured fields (deps:, rules:, Depends:) do not require new syntax — they live inside Python-native docstrings. Many codebases already have docstrings; CodeDNA adds navigation intent to documentation that already exists. The distinction between a standard docstring and a CodeDNA docstring is not syntactic but semantic: the latter must be machine-oriented and navigation-oriented, not merely human-readable.

6.5 Limitations

Sample size. The SWE-bench evaluation uses 5 runs per task at T=0.1, enabling Wilcoxon significance testing. However, N=5 tasks (all from a single Django repository) limits generalisability. Future work should expand to ≥30 tasks across multiple repositories and evaluate on the full SWE-bench suite with real-agent toolchains (Devin-style, Cursor, GPT-tools loop).

Localization vs. resolution. The SWE-bench evaluation measures file localization F1, not end-to-end patch correctness. While localization is a necessary precondition for correct patching, it is a proxy metric. Whether CodeDNA's navigation improvements translate to higher resolution rates remains an open question for future work.

Annotation synchronization. The primary maintenance challenge is not writing annotations (in agentic workflows the cost is near-zero) but ensuring they remain consistent with the code over time. If an agent modifies a function without updating the enclosing rules: or Depends: fields, the annotation becomes misleading — potentially worse than no annotation. This shifts the research problem from production cost to consistency enforcement: pre-commit hooks, CI validation, and agent-side re-annotation obligations are the correct mitigations. A complementary direction is a "CodeDNA Bootstrapper" — a batch LLM pass that injects initial annotations into legacy codebases, preparing them for agentic maintenance. Future integrations should treat annotation drift as a first-class failure mode.

Multi-model evaluation. All 3 models complete. Results are consistent across models (positive Δ on 3–4/5 tasks each) but the SWE-bench subset (5 tasks, all Django) is small and all tasks come from a single repository. Task 13495 shows a consistent negative Δ for Pro and DeepSeek that is not yet explained. Cross-provider runtime differences (function-calling implementations, tool orchestration) introduce variability that is not fully controlled. The results should be replicated on a larger, more diverse task set before generalisability claims can be strengthened.

Bug complexity. The SWE-bench tasks evaluated each require understanding one or two cross-file contracts. Bugs spanning five or more architectural layers may amplify the advantage further — a hypothesis that requires a larger-scale evaluation to test.

Results may represent a lower bound, pending empirical validation of three open questions. The improvements reported here were measured on CodeDNA v0.7 without three extensions that may provide additional gains, but whether they do so remains empirically open. First, whether fine-tuning on CodeDNA corpora improves results: benchmark models are evaluated zero-shot with no prior training on the protocol; a fine-tuned model might treat used_by: and rules: as native navigation signals, but this has not been tested. Second, whether the message: field — agent-to-agent conversational memory — reduces retry rate: each session currently starts from scratch with no knowledge of what previous agents tried or where they failed; whether persistent inter-session memory improves outcomes requires a dedicated experiment. Third, whether cross_cutting_patterns: recovers task-11808-style gains: the proposed v0.8 extension would allow cross-cutting fixes to be guided, but whether this recovers the Δ≈0% observed on cross-cutting tasks is an open empirical question. The current results may therefore represent a lower bound, but this framing itself requires validation.

6.6 Future Work

Agent-to-agent conversational layer (message:). The current agent: field records what an agent did — a write-once narrative entry. A proposed extension for v0.8 adds a message: sub-field that introduces a persistent conversational layer directly in the source file. Unlike rules: (architectural truth, updated in-place), message: entries are tentative observations, open questions, or forward-looking notes left for the next agent. They carry an explicit lifecycle: an observation is either promoted to rules: when confirmed across sessions, or dismissed with a reply that cites the reason. The field operates at both Level 1 (module docstring) and Level 2 (function docstring), preserving sliding-window safety. This extends the inter-agent communication model from a write-once architectural channel to a persistent asynchronous dialogue — multiple agents, multiple vendors, across an arbitrarily long project history, coordinating through the source file itself.

Agent telemetry via git trailers. Git is already immutable, append-only, and diff-complete. A proposed extension embeds AI session metadata directly in commit messages using git trailers — the same convention as Co-Authored-By:, natively recognised by GitHub and standard git tooling. The proposed fields are: AI-Agent:, AI-Provider:, AI-Session:, AI-Visited: (files read during the session, which git does not track natively), and AI-Message:. Combined with the existing .codedna session log and the file-level agent: field, this creates a three-tier audit architecture linked by a shared session_id: git (authoritative diff and metadata), .codedna (navigable session summary for agent planning), and file agent: (sliding-window-safe one-liner). The architecture enables immediate audit queries without external tooling — for example, retrieving all files an agent read in a given session, or comparing navigation behaviour across model providers over the project history.

Cross-cutting pattern support (cross_cutting_patterns:). Task 11808 (__eq__ returning NotImplemented) demonstrates a structural limit of v0.7: cross-cutting concerns — fixes applied to N independent files sharing no common ancestor in the used_by: graph — yield Δ≈0% because the protocol provides no navigation path between them. A proposed Level 0 extension adds a cross_cutting_patterns: field to the .codedna manifest, populated by an agent post-fix as accumulated knowledge: "all classes implementing __eq__ must return NotImplemented for unknown types." This is architecturally distinct from pre-populating a file list for evaluation — the pattern is written by an agent that has already navigated the codebase, not by the annotator preparing the benchmark. Whether this extension recovers the navigation advantage on cross-cutting tasks is an open empirical question.

Verification agents. The annotation synchronization risk identified in §6.5 — annotations diverging from the code they describe — motivates a class of verification agents that periodically cross-check annotations against the actual codebase. Because annotations are structured and machine-readable, this maintenance is itself automatable. Proposed verification types include: navigation correctness (does following used_by: links reach the correct files?), rule compliance (does the code respect its own rules: constraints?), annotation accuracy (do exports: symbols match the actual public API?), and message: lifecycle (are open observations stale or resolved?). Verification agents represent the natural complement to annotation-writing agents — closing the feedback loop between the protocol and the code it describes.

CodeDNA as a self-generating training corpus. The following section describes a speculative future direction that has not yet been empirically validated. A consequential but under-explored property of the protocol is that a CodeDNA-annotated codebase with active agent usage constitutes a self-generating training corpus across three complementary levels of supervision signal.

At the supervised fine-tuning (SFT) level, the append-only agent: field accumulates an ordered, file-scoped narrative of correct navigational decisions. Each line encodes — in natural language — what an agent did, what constraint it discovered, and what it left for the next session. Across thousands of files and sessions, this constitutes a dense demonstration dataset of expert codebase navigation grounded in real task outcomes.

At the preference alignment (DPO/RLHF) level, the git history with AI trailers provides naturally occurring (rejected, chosen) pairs. When Agent A introduces an incorrect annotation or violates a domain constraint, and Agent B subsequently corrects it and updates rules:, the two commits form a labeled preference pair — complete with visited-file lists, session identifiers, and agent-written rationale in AI-Message:. Unlike synthetic preference data, these pairs are grounded in the actual codebase state at the time of each decision, carry the full navigational context via AI-Visited:, and are produced at zero marginal cost during normal development.

At the process reward (PRM) level, the session trace infrastructure already present in the benchmark runner (traces_to_training.py) records the full ordered sequence of tool calls per session. Combined with the binary task outcome (patch applied successfully or not), each trace becomes a labeled reasoning trajectory. Steps that led to a correct outcome receive positive reward; steps on a session that failed can be assigned negative signal, enabling a process-level reward model trained on real agent behaviour rather than on human-generated chain-of-thought.

These three levels are mutually reinforcing. Models fine-tuned on CodeDNA-derived SFT data navigate annotated codebases more accurately, producing higher-quality annotations; higher-quality annotations yield more informative preference pairs; more informative preference pairs improve process reward models that further sharpen navigational policy. This constitutes a data flywheel: each agent session that writes or corrects a CodeDNA annotation improves the training signal available to all future model generations — without requiring additional human labelling effort or dedicated data collection infrastructure. The codebase is simultaneously the environment, the protocol, and the training dataset.

7. Conclusion

We presented CodeDNA, a four-level (Level 0–3) in-source communication protocol whose design uses a biological analogy as a guiding intuition: just as DNA encodes a complete organism blueprint in every cell, CodeDNA ensures that every source file — and every function within it — carries enough architectural context for an LLM agent to navigate correctly without external lookup.

Our SWE-bench multi-model evaluation provides preliminary evidence that in-source architectural context can reduce navigation cost for LLM agents: positive improvements in file localization F1 across all 3 models — Gemini 2.5 Flash +13pp (60% → 72%, p=0.040), DeepSeek Chat +9pp (50% → 60%), Gemini 2.5 Pro +9pp (60% → 69%) — with positive Δ on 3–4/5 tasks per model. These results suggest that CodeDNA functions as a navigation prior that can benefit diverse model architectures, though the small sample size (5 tasks, single repository) limits the strength of generalisability claims.

The results are consistent with the core hypothesis: when cross-file contracts are declared at the point of use, agents can navigate more efficiently without following each contract back to its source. Whether this navigation improvement translates to higher end-to-end bug resolution rates is an important open question that we leave to future work with larger-scale evaluations.

Fine-tuning potential and improvement ceiling. All benchmark results reported here are zero-shot: models follow used_by: links and rules: constraints by general language understanding alone, with no prior training on the CodeDNA protocol. Whether fine-tuning would further improve results is an open empirical question. Three extensions have been proposed that may yield further gains, but each requires dedicated empirical evaluation: (1) the message: field could give each agent session memory of what previous agents tried and where they failed; (2) fine-tuning on CodeDNA-annotated corpora might allow models to treat the structured fields as native navigation signals rather than free text; (3) cross_cutting_patterns: could extend guidance to the class of cross-cutting bugs (e.g., task 11808) where v0.7 currently shows Δ≈0%. Whether any of these extensions yields meaningful gains is an open empirical question that we leave to future work.
