CodeDNA: An In-Source Annotation Standard
for Deterministic Navigation of Multi-File Codebases
by LLM Agents

Fabrizio Corpora
Independent Research · fabrizio@corpora.dev
arXiv preprint · March 2026
Abstract LLM agents that autonomously edit production codebases suffer from context blindness: they modify a function without awareness of cross-file dependencies, producing silent bugs and broken cascades. Existing mitigations — RAG, project-level rules files (CLAUDE.md, AGENTS.md), or long-context injection — impose retrieval latency, maintenance overhead, or infrastructure cost. Recent work (ETH Zurich, 2026) further shows that LLM-generated context files can degrade task success rates by 3% and increase inference costs by over 20%. We present CodeDNA, an in-source annotation standard that embeds dependency graphs, public contracts, agent constraints, and semantic variable conventions directly into source files as structured comments. CodeDNA requires no build step, no external system, and zero retrieval latency. We evaluate CodeDNA with a live agentic benchmark: a Gemini 2.5 Flash agent equipped with read_file, list_files, and grep tools navigates an 11-file Python SaaS codebase to localize a soft-delete bug. The CodeDNA-annotated codebase required 25% fewer tool calls and caused the agent to read 33% fewer files while achieving identical diagnostic accuracy. We argue that in-source annotation is a lightweight, drift-free foundation for reliable LLM-assisted software engineering.

1. Introduction

The adoption of LLM coding agents in production workflows has exposed a class of failure that single-file benchmarks routinely miss: cross-file context blindness. When an agent reads a function in isolation — because it operates in sliding-window mode or because the relevant contract resides in another module — it cannot observe the architectural constraints governing correct behaviour.

SWE-bench [1], the field's principal multi-file benchmark, demonstrated in 2024 that state-of-the-art models could resolve only 1.96% of real GitHub issues. The dominant failure mode was not reasoning quality but navigation cost: finding the right files required too many tool interactions. Subsequent work confirms the pattern: LoCoBench-Agent [2] (2025) measures tool-usage efficiency in long-context engineering workflows and finds systematic over-exploration; PRDBench [3] (2025) reports that agents spend the majority of their token budget on navigation rather than repair.

Current mitigation approaches share a structural weakness: context lives outside the code. CLAUDE.md and CursorRules are session-level files invisible to agents in sliding-window mode. RAG systems introduce retrieval latency and require embedding infrastructure. A 2026 ETH Zurich study [4] found that LLM-generated context files (AGENTS.md) degraded task success by 3% and increased costs by over 20% compared to no context file, while human-written files offered only a 4% gain.

We propose a complementary approach: embed context inside the code, at every level, so that any fragment an agent reads is already self-sufficient. Inspired by biological holography — cutting a hologram in half produces two smaller complete images, not two halves — CodeDNA ensures that a 10-line extract from any file carries enough structure for an agent to navigate safely.

Contributions:

(2) A live agentic benchmark using Gemini 2.5 Flash with function-calling tools on a real on-disk codebase.

2. Related Work

2.1 Multi-File LLM Benchmarks

SWE-bench [1] evaluates LLMs on 2,294 real GitHub issues across 12 Python repositories. It established that multi-file coordination is the principal bottleneck: Claude 2 solved only 1.96% of tasks at launch. LoCoBench-Agent [2] extends this to interactive, long-context workflows and measures tool-usage efficiency explicitly — the metric most aligned with CodeDNA's contribution. PRDBench [3] uses Product Requirement Documents as task descriptions and an Agent-as-a-Judge paradigm; it reports to AAMAS 2026.

HumanEval [5] (Chen et al., 2021) measures single-function code synthesis from docstrings. Its pass@k metric is standard but does not capture cross-file navigation, making it a complementary rather than competing benchmark for our purposes.

2.2 Context Management for LLM Agents

Retrieval-Augmented Generation [6] retrieves relevant chunks from a vector store at inference time. Applied to code, retrieval quality depends on embedding similarity, which may not align with architectural relevance. Rules files (CLAUDE.md, CursorRules) impose project-level conventions but are invisible to sliding-window agents and require manual maintenance. The ETH Zurich study [4] provides the strongest evidence against the external-file approach: even carefully crafted files offer marginal gains, and LLM-generated ones actively degrade performance.

MemGPT [7] maintains a compressed memory across turns. This addresses recall of prior context but not the structural dependency problem: a history of prior reads does not tell an agent what to read next. Knowledge graph–based approaches (Tongyi Lingma, 2025–2026) use graph traversal to expose cross-module call relationships but rely on external infrastructure and dynamic analysis.

2.3 Code Annotation Standards

Documentation standards (docstrings, JSDoc, Doxygen) focus on human readability and API documentation generation. They do not encode cross-file navigation semantics or agent-specific constraints. CodeDNA differs by designing annotations as a machine-readable navigation protocol rather than a documentation format.

3. The CodeDNA Annotation Standard v0.5

CodeDNA operates at three levels, each independently useful.

3.1 Level 1 — Manifest Header

A structured comment block that must be the first content in every file. It encodes, in approximately 70 tokens, the full architectural position of the file.

# === CODEDNA:0.4 ===========================
# FILE:           orders/orders.py
# PURPOSE:        Order lifecycle management
# CONTEXT_BUDGET: always
# DEPENDS_ON:     db/queries.py :: execute()
#                 users/users.py :: get_user()
# EXPORTS:        get_active_orders() → list
# REQUIRED_BY:    analytics/revenue.py
# RULES:          User system uses soft delete.
#                 See users/users.py :: Rules.
# LAST_MODIFIED:  initial implementation
# ==========================================

The CONTEXT_BUDGET field (always / normal / minimal) enables a Planner Manifest-Only Read protocol: an orchestrating agent reads only the first 14 lines of each file, filters by budget value, and builds a full dependency map before deciding which files to open in full. A 50-file codebase can be mapped in approximately 3,500 tokens.

3.2 Level 2 — Inline Hyperlinks

Agents frequently extract 50–100 lines from the middle of a file, bypassing the Manifest Header. Level 2 annotations embed navigation cues at the function level.

def get_active_orders():
    # @REQUIRES-READ: users/users.py :: delete_user()
    # @MODIFIES-ALSO: analytics/revenue.py
    return execute(
        "SELECT * FROM orders WHERE status != 'cancelled'"
    )

The four annotation tags are:

• @SEE — recommended context; read when uncertain.

• @REQUIRES-READ — mandatory; agent must read target before writing.

• @MODIFIES-ALSO — cascade; agent must update target symbol in same changeset.

• @BREAKS-IF-RENAMED — identity is load-bearing; must not rename without updating all callers.

3.3 Level 3 — Semantic Naming

Variable names encode type, shape, domain, and origin in a compound identifier. Format: <type>_<shape>_<domain>_<origin>.

# ❌ Standard — requires full trace to understand
data  = get_users()
price = request.json["price"]

# ✅ CodeDNA — self-documenting in any 10-line window
list_dict_users_from_db  = get_users()
int_cents_price_from_req = request.json["price"]

4. Benchmark Design

We measure navigation efficiency — how many tool interactions an autonomous agent requires to correctly localize the root cause of a bug in a multi-file codebase. Accuracy is held constant; cost is the variable of interest.

4.1 Task and Codebase

We generated two versions of an 11-file Python SaaS application (Flask server, orders, users, authentication, analytics, dashboard, payments, notifications, configuration, and database layer):

• Control: standard Python source with no architectural annotations.

• CodeDNA: identical logic, every file annotated per the v0.5 standard. Annotations describe architecture only — they contain no hints about the bug.

Planted bug. The user system implements soft delete: delete_user() sets users.deleted_at = NOW() but does not remove the row. orders/orders.py::get_active_orders() does not join on the users table; it returns all non-cancelled orders regardless of user deletion status. Orders belonging to deleted users thus inflate the revenue dashboard.

The ground-truth root-cause file is orders/orders.py. The fix requires joining with the users table and filtering deleted_at IS NULL.

Why this task is discriminating. The bug cannot be located by inspecting any single file in isolation. The agent must: (1) understand that the user system uses soft delete, defined in users/users.py; (2) recognise that orders/orders.py does not account for this contract; (3) trace the call chain views/dashboard.py → analytics/revenue.py → orders/orders.py.

In the Control version the soft-delete contract is undocumented outside users/users.py. In CodeDNA, orders/orders.py explicitly declares in its RULES field: "User system uses soft delete — see users/users.py :: Rules."

4.2 Agent Implementation

We implemented an autonomous agent using Gemini 2.5 Flash (temperature 0, deterministic) with three function-calling tools:

• read_file(path) — reads the complete content of a file on disk.

• list_files(directory) — recursively enumerates files in a directory.

• grep(pattern, directory) — searches for a text pattern across all Python files.

The agent receives a natural-language bug description (Appendix A) and must use these tools autonomously until it can produce a final answer. No file contents are pre-loaded into the prompt. The agent runs to completion (max 20 turns) or until it stops calling tools.

Metrics collected: total tool calls, read_file calls, list_files calls, grep calls, conversation turns, and found_correct (boolean).

5.1 Navigation Path Analysis

Control agent: (1) list_files(""); (2) read_file("analytics/revenue.py"); (3) read_file("orders/orders.py") — observes no user filtering; (4) read_file("users/users.py") — manually verifies soft-delete contract; (5) final answer.

CodeDNA agent: (1) list_files(""); (2) read_file("analytics/revenue.py") — manifest declares dependency on orders/orders.py; (3) read_file("orders/orders.py") — RULES field states soft-delete contract inline; (4) final answer.

The eliminated read (users/users.py) represents the class of inferential lookup that CodeDNA prevents: the agent needed a third file not because it was directly relevant, but to verify a contract already declared at the point of use.

6. Discussion

6.1 Positioning Against Related Work

CodeDNA is complementary to SWE-bench [1]: where SWE-bench measures whether an LLM resolves a bug (output quality), CodeDNA addresses how efficiently the agent navigates to the bug location (navigation cost). These are orthogonal dimensions of agent capability.

The ETH Zurich finding [4] that LLM-generated context files degrade performance precisely motivates our approach: if external context files cause over-exploration, the solution is not better files but embedded context that arrives with the code the agent was already going to read.

6.2 Why In-Source Annotation Works

The mechanism is simple: when the contract governing a function is declared at the function, capable agents do not need to retrieve it from its original definition. The RULES field in orders/orders.py — "user system uses soft delete, see users/users.py" — gave the CodeDNA agent sufficient information to understand the bug without opening users/users.py. The annotation propagated the contract downstream to where it was violated.

6.3 Annotation Overhead and Maintenance

Co-location eliminates drift: the manifest lives in the same file as the code it describes. Tooling enforces updates: tools/validate_manifests.py checks field consistency and a pre-commit hook blocks commits with empty LAST_MODIFIED fields. The per-file annotation cost is 8–12 comment lines.

6.4 Limitations

Single run. Temperature 0 ensures determinism, but one data point limits statistical confidence. Future work should vary the bug type, codebase size, and model.

Bug complexity. The planted bug requires understanding exactly one cross-file contract. More complex bugs may amplify or diminish CodeDNA's advantage.

Annotation quality. The benchmark uses hand-written, high-quality annotations. Real-world adoption may produce variable quality; LLM-assisted annotation generation is a promising direction.

Single model. Results are specific to Gemini 2.5 Flash. Models with longer effective context windows may show smaller differences.

7. Conclusion

We presented CodeDNA, a three-level in-source annotation standard that embeds dependency graphs, public contracts, agent constraints, and semantic naming conventions directly into source files. The standard requires no external infrastructure, no build step, and no retrieval system.

Our live agentic benchmark — a Gemini 2.5 Flash agent with function-calling tools on a real on-disk codebase — demonstrates a 25% reduction in tool calls and a 33% reduction in file reads for a representative bug-localization task, with identical diagnostic accuracy.

The result validates the core hypothesis: when cross-file contracts are declared at the point of use, capable agents can reason correctly without following the contract back to its source.
